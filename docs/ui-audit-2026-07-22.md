## 1. 设计读取

> 读取为：**个人专注工具的三页工作台**，面向把它当日常主力工具的单一深度用户，用暖纸质编辑语言，倾向 Notion / Stripe 那一支「暖色高端 + 内容优先」。

不是营销页，不是仪表盘产品。功能契约：

| 页面 | 3 秒内必须获得 |
|---|---|
| 计时页 | 还剩多久、在做什么、现在能按什么 |
| 清单页 | 今天要做几个、预算还剩多少 |
| 统计页 | 我这段时间到底怎么样 |

## 2. 三拨盘取值

分页给值，三个页面的性格本来就不同。

| 拨盘 | 计时页 | 清单页 | 统计页 | 说明 |
|---|---|---|---|---|
| 视觉冒险度 | 4 | 5 | 4 | 版式保持可预期。每天用的工具，不是作品集。现状约 5，基本对 |
| 动效强度 | **2** | 3 | 2 | 计时是长时间凝视的界面，多余动效即噪音。现状名义 4，但是**没设计过的 4** |
| 信息密度 | 3 | 6 | **7** | 统计页本来就该密，问题是密得没节奏 |

用户可随时说「更稳一点 / 动效再少 / 统计页太密」来调整。

## 5. 建议的执行顺序（视觉复核后修订）

按「少而果断」的原则，不要一次全上。

| 拨次 | 内容 | 理由 |
|---|---|---|
| **第一拨** | P0-1、P0-3 ~ P0-6 | 一次 token 层改动 + 几处删除，收益最大、diff 最小，改完整个界面立刻变清楚。复核确认这几处的严重程度与静态判断一致 |
| **第一拨半** | P0-2 | 复核发现比原判断更严重：月视图日期轴不是"偏小"，是**完全重叠不可读**（见 §6 表格第 4/6 项）。改字号不够，需要同时做"隔位显示"逻辑，成本从"中"上调，建议单独一个 commit，不要和纯 token 改动混在一起 |
| **第二拨** | P0-7、P1-8、P1-9、P1-10 | 触屏可达性 + 动效系统。P0-7 复核**强确认**且比预期更绝对：桌面无 hover 状态下、移动端窄屏下，活动清单顶层任务与全部子任务的操作按钮完全不可见也点不到，唯一的例外是点进"编辑标题"模式会露出一套不完整的按钮（确认/取消），不能替代详情/完成/删除 |
| **第三拨** | P1-12、P2-14 ~ P2-16 | 死代码清理 + 文案。独立 commit，方便单独回退。P2-14 复核确认：统计页副标题原文是"由 Task、Session、Event、EnergyRecord 与 DayPlan 实时派生"，卡片里直接出现 `standard focus`/`extraFocus`/`discarded focus`/`shortBreak`/`longBreak`/`workEnded 豁免`/`Session.actualDuration` 等未翻译的 schema 术语，确认是"读起来像 schema dump" |

**建议本轮不做**：

- **P1-11（拖拽反馈）** — 要新增 React state，超出「少而果断」
- **P1-13（层级合并）** — 大面积重构，风险与收益不成比例

每拨按页面拆原子 commit。每次 commit 前展示 `git diff --stat` + 建议 commit message，用户确认后才提交，禁止自动 push。每轮改完必须跑 `npm run test:run` + `typecheck` + `build` 并如实报告。

## 6. 视觉复核待办（已完成，结论如下）

用 Claude in Chrome 实测，dev server `localhost:5173`。工具限制：`resize_window` 对已打开的 tab 不生效（`window.innerWidth` 实测不变），必须先 `resize_window` 再新建 tab 导航，viewport 才会真的变窄；本次复核用新 tab 在 500px 宽（约等于 560px 断点内侧）测试。桌面全宽用 1440×900（渲染约 1200 CSS px）。

| # | 页面 / 状态 | 复核什么 | 关联 | **结论** |
|---|---|---|---|---|
| 1 | 清单页 · 桌面全宽 | `--muted` / `--muted-2` 真实可读性；子任务名颜色；底部居中说明文字 | P0-1、P2-16 | **确认**。空态文案「清单里暂时没有活动。」、底部说明文字肉眼明显发虚；新建子任务后子任务名颜色显著浅于顶层任务名（几乎融入背景），与 2.3:1 的计算值吻合 |
| 2 | 清单页 · 桌面 · **悬停在任务行上** | `.atr-actions` 悬停显形的实际效果 | P0-7 | **确认**。不悬停时按钮完全不可见（非仅弱化）；悬停后正常显示 详情/完成/删除 |
| 3 | 计时页 · 桌面 · 空闲态 | 任务名字体是否为宋体；圆环；导航选中态左侧竖条 | P1-10、P0-4 | **确认**。任务名「计划准备」肉眼可辨的衬线体（宋体），与休息建议等黑体正文不一致；导航选中项左侧橙色竖条清晰可见 |
| 4 | 统计页 · 桌面全宽 | 信息密度、8px/9px 日期轴、标签拥挤、整体节奏 | P0-2、拨盘 | **比预期更严重**。日视图日期轴（单点）尚可辨；**月视图日期轴完全重叠成一团乱码**（"07-107-207-3…"连在一起，无法分辨任何一个日期）。不是"偏小"，是不可用 |
| 5 | 清单页 · **560px 区间（实测 500px）** | `.atr-actions` 是否完全不可见 | P0-7 | **确认，且更绝对**。顶层任务与子任务的操作按钮组无 hover 时不可见；点击进入"编辑标题"模式会显形一套不同的按钮（确认/取消改名），但不包含详情/完成/删除，不能替代 |
| 6 | 统计页 · **560px 区间** | 小字号在窄屏的实际表现 | P0-2 | 单日视图下 KPI 卡片单列堆叠，数字与标签本身可读；未在窄屏下复现月视图图表（元素在首屏以下），日期轴拥挤问题预期与桌面一致或更差，风险判断不变 |
| 7 | 任意页 · 任务详情弹窗 | `.task-detail` 左侧竖线、弹窗排版 | P0-4 | 弹窗本体（`dialog` 根层）未观察到左侧竖线；`:1456` 的 `border-left: 2px` 位置未在本轮弹窗内容中出现，需要改动时对照源码行号而非仅凭这次截图判断，不排除是弹窗内某个子状态才触发 |
| 8 | 交互过程 | transition 是否产生明显糊感或多余干扰 | P1-8、P1-9 | 未做专项高帧率录制复核，本轮操作（点击、悬停、弹窗开关）未观察到明显掉帧；维持原判断（`transition: all` + 缓动无系统仍是隐患，只是未被目测直接证实） |
| 9 | 全部 | 静态审计没发现的新视觉问题 | — | 新发现：清单页顶层任务名下的子任务列表在**任务详情弹窗内不显示**（子任务只在弹窗背后的主列表里出现），这是既有产品行为而非本轮改动范围，仅记录，不建议本轮处理 |

**复核方法的一个坑**：`resize_window` 只有配合"先设置尺寸、再在同一 tab 里 `navigate`/新建 tab"才会真正改变 `window.innerWidth`；对已经渲染完的页面调用它不会触发响应式断点重排。以后要测断点，记得走这个顺序。
